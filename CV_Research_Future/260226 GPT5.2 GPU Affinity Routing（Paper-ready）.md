## 摘要

大规模 MoE 推理在不同工作负载下呈现出两类主导瓶颈：在 **EP decode 小 batch** 的 memory-bound regime 中，端到端延迟主要由每层 **被激活的 expert replica 数量**决定；而在 **prefill / 大 token 批量 / 多节点** 的 communication-bound（或 mixed）regime 中，跨节点 All-to-All 产生的通信开销会成为主要瓶颈。已有工作 METRO 通过“平衡激活专家数而非 token 数”，有效优化了前者，但对后者（跨节点通信成本主导场景）覆盖不足。

本文提出一种 **Topology-aware MoE routing**：在不改变模型参数的前提下，将“token 在上一层所在设备/节点”作为状态，引入拓扑感知通信成本，并在保留语义路由质量的约束下，对候选 expert 进行 **two-stage re-rank**（先语义 top-$M$，再在候选内结合通信代价与负载守护选 top-$k$）。该策略可实现：

- 在 multi-node / prefill-heavy 服务中显著降低 cross-node bytes 与通信时间；
- 通过候选集约束与质量守护，控制路由偏置带来的质量退化风险；
- 在 OR 视角下可建模为带拓扑成本的在线随机网络控制问题，并用 Lyapunov drift-plus-penalty 给出稳定性与最优性权衡。

## 1. 背景与问题界定

### 1.1 两个 regime：为什么要把 METRO 和 Topology/Communication-aware routing 分开讲

**结论（paper-friendly）**：MoE serving 的瓶颈随 regime 变化。

- **Memory-bound decode（小 batch，比如 1–32 tokens/GPU）**：FFN 常由权重加载主导。
- **Communication-bound / mixed prefill（大 token 批量、多节点、跨机互联）**：All-to-All 的带宽与时延成本上升，cross-node 通信变为可优化对象。

这意味着：

- METRO 的核心贡献是优化 **memory-bound decode**（减少激活 expert replicas → 少读权重 → FFN 变快）；
- Topology-aware / Communication-aware routing 的目标应明确定位在 **multi-node / prefill-heavy / bandwidth-dominant** 场景，用“减少跨节点通信（cross-node bytes / migrations）”来获得收益。

### 1.2 METRO Figure 11 的关键结论

METRO（arXiv:2512.09277）在 8×A100 decode 的 runtime breakdown（Figure 11）显示：

- FFN：EPLB 约 ~350μs，METRO 约 ~269μs（下降 ~81μs）；
- Communication：两者均约 ~55μs（统计上近似不变）；
- 额外开销：全局 top-k 与 METRO 算法开销约数十 μs，但被 FFN 的降低抵消。

**解释**：在该设置下单层通信量很小，通信时间主要被 NCCL 启动/同步固定开销主导，因此“bytes 减少”未必反映为“time 减少”。

> 因此本文不以“单机 decode 的通信时间下降”为主要卖点，而以“多节点 prefill 的跨节点通信下降”为主。

## 2. 机制：为什么 decode 下 FFN 取决于激活 expert 数

decode 阶段每个 expert 接收到的 token 数 $n_e$ 很小，GEMM 的计算量随 $n_e$ 缩放，但权重读取量近似不随 $n_e$ 变化。Roofline 视角下，当 $n_e \ll d_{ffn}$ 时，operational intensity 极低，进入 memory-bound。

一个可直接写进论文的近似关系是：

$$
T_{\text{FFN}} \approx \frac{|\mathcal{A}| \cdot W_e}{\mathrm{BW}_{\mathrm{HBM}}}
$$

其中 $|\mathcal{A}|$ 为该层被激活的 expert replicas 数，$W_e$ 为单个 expert 权重大小。

这也解释了：METRO 通过将同一 expert 的 token 尽量集中到少数 replicas 上，从而减少 $|\mathcal{A}|$ 并降低 FFN 时间。

## 3. 我们的方法：Topology-aware routing with Two-stage Re-rank

### 3.1 设计原则（落地与质量的硬约束）

为保证工程可落地与质量不崩，我们遵循两条原则：

1. **语义候选集合不动**：先按原 router logits 选出 top-$M$ 候选（$M>k$）。
2. **只做候选内重排**：在 top-$M$ 内加入拓扑代价与负载守护项，选最终 top-$k$。

这使得本方法更像“候选集 re-rank / tie-break”，而不是“把不相关 expert 强行塞进 top-k”。

### 3.2 实现 contract

**输入（每个 MoE 层）**：

- token 隐状态 $h_t$（由模型给出）；
- 原始 router logits $\ell_{t,e}$；
- token 当前所在设备/节点 $s_t$（上一层 dispatch 后的位置，或路由器可追踪的 rank id）；
- expert placement 映射 $\pi(e,r)$：expert $e$ 的 replica $r$ 所在 device/node；
- 拓扑成本函数 $\mathrm{CommCost}(s, d)$：从 source $s$ 到目标 device/node $d$ 的通信代价（可按同 GPU/同节点/跨节点离散化）。

**输出**：

- 对每个 token 的 top-$k$ experts（以及对应的目标 replica / rank），满足 capacity / load guard。

**可观测性假设**（工程上通常成立）：

- placement 映射在一次权重加载期间近似静态；
- $s_t$ 可由运行时追踪（无需真实 GPU FIFO 队列信息）；
- 负载指标可由路由阶段统计（per-rank 分配 token 计数），不依赖硬件队列长度。

### 3.3 Two-stage 具体形式

### Stage A：语义候选

对每个 token $t$：

$$
\mathcal{E}_t = \mathrm{TopM}(\ell_{t,e})
$$

### Stage B：候选内拓扑感知 + 负载守护重排

在候选内对每个 expert/replica 计算重排分数：

$$
\mathrm{Score}(t,e,r) = \ell_{t,e} - \eta \cdot \mathrm{CommCost}(s_t,\pi(e,r)) - \lambda \cdot \mathrm{LoadPenalty}(\pi(e,r))
$$

并输出：

$$
\mathrm{TopK}_{(e,r) \in \mathcal{E}_t \times \mathcal{R}(e)}\, \mathrm{Score}(t,e,r)
$$

其中：

- $\eta$ 控制“性能优先/通信优先”；
- $\mathrm{LoadPenalty}$ 用于防止局部热点（可用路由阶段的 token 预算或滑窗计数近似），避免 tail-latency 恶化；
- 该形式与“直接给本地 expert 加 bias”相比，具备可控的质量风险边界：因为候选集仍由语义 logits 产生。

### 3.4 与 METRO 的互补与组合（论文 story）

- **decode（memory-bound）**：采用 METRO-like 目标（最小化 activated replicas），或直接启用 METRO。
- **prefill / multi-node（comm/mixed）**：启用本文的 topology-aware rerank（最小化 cross-node bytes）。
- **统一控制**：以负载与通信代价的权重参数（如 $\eta$）随工作负载动态调整，实现一个在两种 regime 间切换的统一策略。

## 4. OR 建模视角（MS 版本可扩展出的“模型—策略—定理”骨架）

### 4.1 系统建模（状态/动作/成本）

- **状态**：$(c_t, s_t, \mathbf{q}_t)$，其中 $c_t$ 是 token 类别/语义簇（可由路由器隐式表示），$s_t$ 是 token 当前所在 device/node，$\mathbf{q}_t$ 是 per-device 的“分配压力/预算使用”统计。
- **动作**：在候选集内选择目标 $(e,r)$，等价于选择目标设备 $d=\pi(e,r)$。
- **即时成本**：通信代价 $\mathrm{CommCost}(s_t,d)$ + 负载代价 $f(q_d)$，并以“候选集约束”确保语义质量。

### 4.2 在线控制（drift-plus-penalty 的可实现形式）

可写成：

$$
(e^*,r^*) = \arg\max_{(e,r)\in \mathcal{E}_t\times\mathcal{R}(e)} \left\{\ell_{t,e} - \eta\,\mathrm{CommCost}(s_t,\pi(e,r)) - \lambda\,f(q_{\pi(e,r)})\right\}
$$

该形式直接对应实现中的 rerank 评分。

> 论文写作建议：在 MS 版本中给出稳定性与最优性 gap 的标准结论（在有界到达与有界成本下），把工程策略“抬升”为可证的在线控制。

## 5. 实验矩阵（paper-ready）

### 5.1 研究问题（RQ）

- **RQ1**：在 multi-node prefill-heavy 场景下，topology-aware rerank 能否显著降低 cross-node bytes 与通信时间？
- **RQ2**：性能提升与质量损失之间的 Pareto 曲线如何（扫 $\eta,\lambda,M$）？
- **RQ3**：负载守护项是否能降低 tail latency（p95/p99）并避免热点？
- **RQ4**：与 METRO 的组合是否能覆盖 decode 与 prefill 两种 regime？

### 5.2 场景与变量

| 维度 | 设置（建议） | 目的 |
| :--- | :--- | :--- |
| 拓扑 | 单节点 NVLink vs 2+ 节点 IB | 验证“跨节点才显著” |
| 阶段 | prefill（长上下文） vs decode（小 batch） | 区分两种瓶颈 |
| batch/token | 例如 256/1024/2048 prefill tokens | 放大通信量 |
| 路由 | baseline（原始 top-k）、Naive topology-biased routing（全局偏置）、Two-stage rerank（本文） | 比较安全策略 |
| 守护项 | 无守护 vs load penalty | 观察 tail 与稳定性 |

### 5.3 指标（必须覆盖）


**通信层面（核心证据链）**：

- cross-node bytes / layer（或 per forward）；
- node-internal bytes / layer；
- collective 次数与 message size 分布。


**性能层面**：

- TTFT / TPOT；
- throughput（tokens/s）；
- p95/p99 tail latency。


**质量层面**：

- perplexity 或任务指标（至少一条），并给出与性能的 Pareto 曲线。

### 5.4 消融（最小但够用）

- $M$（候选集大小）：$2k,4k,8k$；
- $\eta$（通信权重）：对数刻度扫描；
- $\lambda$（负载守护权重）：0 与若干正值对比；
- 拓扑 cost：同 GPU / 同节点 / 跨节点三档。

## 7. 附：为什么不直接“对全量 logits 加 bias”

直接对所有 experts 的 logits 加“全局拓扑/通信偏置”（topology/communication bias）虽然实现最简单，但存在两类风险：

1. **质量风险**：可能将语义不匹配但本地的 expert 推入 top-k；
2. **稳定性/尾延迟风险**：局部偏置叠加后可能形成热点，触发 capacity 溢出或负载不平衡，p99 变差。

Two-stage rerank 把动作空间限制在语义候选集上，并允许显式加入负载守护项，因此更适合作为“可上线/可投稿”的主方案。

## 8. 潜在风险与客观评估（含可执行对策）

本节总结两类核心担忧，并给出与当前业界/学界 MoE serving 实践一致的处理原则：**语义先决、成本后调（semantic feasibility first, cost-aware tie-breaking second）**。换句话说，拓扑/通信项不应“压过语义路由”，而应主要在“语义候选之间差距接近”时发挥作用。

### 8.1 Topology 如何打分？遇到“拓扑好但质量差 / 拓扑差但质量好”如何均衡？

**客观评价**：这是 topology-aware routing 的核心矛盾，但在本文的 **two-stage rerank** 框架下，它是一个可控权衡问题，而不是不可落地的拍脑袋决策。

**推荐做法（最小可落地）**：

1. **动作空间约束（candidate constraint）**：只在语义 top-$M$ 候选集合内重排。$M$ 决定“给拓扑让路的空间”——$M$ 太小会让拓扑项失效，$M$ 太大则会把语义边缘 expert 引入而增加质量风险。经验上可从 $M \in \{2k, 4k\}$ 起步，并以 Pareto 曲线确定上限。
2. **拓扑成本离散化（topology cost tiers）**：以可解释的三档成本函数作为起点：

  $$
  \mathrm{CommCost}(s,d) \in \{0,\;\gamma_{\text{intra-node}},\;\gamma_{\text{inter-node}}\}
  $$

  其中 $s$ 为 token 当前所在 device/node，$d$ 为候选目标 device/node。
3. **语义差距门控（margin-gated rerank）**：当语义差距很大时不应为了拓扑替换 expert；当语义差距很小时才允许拓扑/负载主导。令 $e_1$ 为语义 top-1 expert，定义

  $$
  \Delta_{t,e} = \ell_{t,e_1} - \ell_{t,e}.
  $$

  只在 $\Delta_{t,e}\le \tau$ 时显著启用拓扑项，或使用软门控 $w(\Delta)$：

  $$
  \mathrm{Score}(t,e,r)=\ell_{t,e} - w(\Delta_{t,e})\,\eta\,\mathrm{CommCost}(s_t,\pi(e,r)) - \lambda\,\mathrm{LoadPenalty}(\pi(e,r)).
  $$

**直观效果**：

- “拓扑好但质量差”：语义差距大 $\Rightarrow w(\Delta)$ 近似为 0 $\Rightarrow$ 不会为了拓扑选择它。
- “拓扑差但质量好”：语义差距大 $\Rightarrow$ 仍优先质量好 expert，仅承担必要通信成本。

### 8.2 rerank 是否会导致输出质量下降过大？

**客观评价**：质量风险真实存在（推理时改变训练得到的 gating 分布可能导致路由偏移），但在“候选内重排 + 语义差距门控 + 小幅偏置”的设计下，风险通常可被压到可接受区间。真正危险的设置是“强偏置 + 脱离语义候选集”。

**三层质量护栏（建议写入实现与实验章节）**：

1. **候选集护栏**：只在 top-$M$ 内重排（本文已采用）。
2. **语义差距护栏**：使用 margin-gating（上节），语义优势明显时不重排。
3. **退火/预算与回退机制**：


- 对“重排比例”或“跨节点迁移率”设置上限（超过则降低 $\eta$ 或禁用 rerank）；
- 当质量监控指标（例如 PPL 或任务指标）劣化超过阈值时自动回退到 baseline routing（failsafe）。

**验证方式（必须量化，而非主观判断）**：

- 画出 $\eta$（以及 $M,\tau$）扫描下的 Pareto 曲线：
  - 横轴：cross-node bytes / layer 或通信时间；
  - 纵轴：TPOT/throughput；
  - 叠加：PPL/任务指标与 p95/p99 tail。

该曲线将直接回答“是否过大”的问题：如果收益主要来自在“语义近似等价”的候选之间做同节点/同 replica 的选择，通常可以以较小质量代价换取可观通信收益；反之若必须频繁替换语义强优势 expert，则质量会快速恶化。
